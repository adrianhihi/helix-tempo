🧬 Helix × Tempo — Judge Q&A

Cheat Sheet · Tempo × Stripe HIIT Hackathon · March 19, 2026

🎯Product & Vision6 questions
LIKELYWhat does Helix actually do? One sentence.›
"Helix wraps any AI agent's payment calls with a self-repair engine — when a payment fails, it automatically classifies the failure, generates repair candidates, picks the best fix, executes it, and retries. Every repair is stored so the same failure never happens twice."
LIKELYWhy not just add retry logic? What's different about Helix?›
"Retry logic is dumb — it just waits and tries again. Helix is intelligent repair. A 'payment-insufficient' error doesn't need a retry, it needs a currency swap. A 'session expired' needs a session renewal, not a retry. PCEC classifies the failure type, generates 2-3 specific repair strategies, scores them by cost and speed, and executes the best one. Then it stores the fix as a Gene Capsule so next time it's instant. Retry logic learns nothing."
Key phrase: "Retry is a hammer. PCEC is a surgeon."
LIKELYHow is this different from Stripe Smart Retries?›
"Stripe Smart Retries work at the card network level — they retry the same charge at different times. Helix works at the agent infrastructure level — it doesn't retry the same action, it repairs the underlying problem. If your agent doesn't have enough USDC, Stripe retrying won't help. Helix sees the insufficient balance, swaps EURC→USDC via Tempo's DEX, and then pays. Different layer, different capability."
MEDIUMWhy build on Tempo specifically? Could this work on any chain?›
"The PCEC repair engine is chain-agnostic in principle. But Tempo gives us three things no other chain has: 1) MPP is the native payment protocol with structured error codes — we parse those directly. 2) Native DEX for instant currency swaps as repair actions. 3) Batch transactions + payment lanes for batch repair scenarios. We're not 'can run on any chain' — we're deeply integrated with Tempo's specific features."
Judges WANT to hear Tempo-specific integration, not chain-agnostic. You're building FOR Tempo.
MEDIUMWho's the customer? Developers? Agents? Companies?›
"Developers who build AI agents that make payments. Two lines of code: import { wrap } from '@helix/tempo' and const safe = wrap(myAgent). That's it. The developer's agent code doesn't change at all. Helix is a transparent proxy — the caller doesn't even know it's there until something fails."
DEEPWhat's the Gene Map? How does it work?›
"Gene Map is a local SQLite database that stores every successful repair as a 'Gene Capsule' — a record of: what failed, how it was fixed, and how fast. Next time the same failure pattern appears, PCEC skips candidate generation and evaluation, and applies the known fix instantly. In our benchmark, 9 out of 14 failures were resolved this way — instant, zero-cost repair from memory. That's the network effect: more failures fixed → more genes → faster repairs → more value."
🔧Technical Deep Dives7 questions
LIKELYAre these 12 failure scenarios real? Or made up?›
Scenarios 1-6 are directly from the MPP protocol spec: The MPP protocol at mpp.dev/protocol defines exactly these error codes — payment-insufficient, invalid-challenge, method-unsupported, verification-failed, malformed-credential. These are in the IETF draft, RFC 9457 structured errors. Scenario 5 (batch revert) is from Tempo's docs which say "if any operation fails, the entire batch reverts."

Scenarios 7-8 are real platform risks: Any DEX has slippage reverts (scenario 7). Tempo's TIP-403 compliance registry can block transfers (scenario 8).

Scenarios 9-10 are vision items: Cascade failure is based on Galileo AI research showing 1 compromised agent affects 87% of downstream decisions in 4 hours. Off-ramp failure is confirmed by Transak as "the primary failure mode" in stablecoin payroll.
"Scenarios 1-6 map directly to MPP's error code table in the IETF spec. 7-8 are real Tempo platform risks. 9-10 are vision items backed by industry research. We tagged them honestly as LIVE, MOCK, and VISION."
LIKELYWhat's PCEC? Walk me through how it works.›
"PCEC = Perceive, Construct, Evaluate, Commit. Four steps: P — Perceive: Parse the MPP 402 error response, classify it. 'payment-insufficient' → category: balance, severity: low. C — Construct: Generate 2-3 repair candidates. For balance: swap alt stablecoin ($0.001, 0.5s), top-up from reserve ($0, 2s), reduce request amount ($0, instant). E — Evaluate: Score each candidate 0-100 by speed, cost, and historical success. Swap wins at score 96. K — Commit: Execute the swap via Tempo DEX, retry the payment, store the fix as a Gene Capsule. Next time → instant."
Walk through with hand gestures: P (point to eyes), C (spread fingers = options), E (thumbs up/down), K (fist = execute).
MEDIUMHow do you parse MPP errors? What format?›
"MPP uses RFC 9457 Problem Details — structured JSON responses with a 'type' field like 'https://paymentauth.org/problems/payment-insufficient'. We parse the type URI to extract the error code, then classify by category. The error space is bounded — MPP defines exactly 7 error codes. We handle all 7 plus 3 additional Tempo-specific ones (batch revert, DEX slippage, compliance block)."
MEDIUMWhy is the error space bounded? How do you know you cover everything?›
"MPP's error codes are defined in an IETF spec — there are exactly 7: payment-required, payment-insufficient, payment-expired, verification-failed, method-unsupported, malformed-credential, invalid-challenge. We added 3 from Tempo's specific features (batch revert, DEX, compliance). The repair strategies are also deterministic — for each category, we generate a fixed set of 2-3 candidates. And Evaluate scores by math (speed + cost + requirements), not AI. The whole pipeline is enumerable. That's what makes it reliable."
This is your strongest technical argument. Error space = finite. Repairs = deterministic. Scoring = mathematical. No hallucination risk.
MEDIUMIs the retry safe? What about double-charging?›
"MPP has idempotency built into the protocol. The Payment credential contains a challenge ID — if you retry with the same credential, the server recognizes it and doesn't charge again. For scenario 6 (service down after payment), we retry with the existing Payment-Receipt header — the server sees the receipt is valid and returns the resource without a new charge. This is by design in the MPP spec."
DEEPHow does this relate to SAGA pattern in distributed transactions?›
"PCEC is SAGA-aware. For cascade scenarios like A→B→C where C fails, we execute compensating transactions in reverse order — classic SAGA pattern. But PCEC goes further than pure SAGA: it doesn't just rollback, it generates multiple repair candidates and picks the optimal one. Sometimes the best fix isn't a refund — it's rerouting through agent D. SAGA gives you undo. PCEC gives you undo OR fix OR reroute, scored by cost and speed."
DEEPWhat happens when PCEC can't fix something?›
"It fails honestly. In our benchmark, 2 of 14 failures were unrecoverable — a task that genuinely needed 200K tokens (context overflow beyond repair) and a persistent server outage (3 retries exhausted). PCEC marks these as 'unrecoverable', stores a Gene noting the pattern, and propagates the original error to the caller. We don't hide failures. We fix what we can and report what we can't."
The 2 unrecoverable failures are your CREDIBILITY play. Judges trust teams that know their limits.
📊Benchmark & Data6 questions
LIKELYExplain the 68% → 90% number. How did you get it?›
"We ran 50 HumanEval coding tasks with GPT-4o-mini. We injected infrastructure failures at realistic rates using a fixed seed — 8% rate limit, 5% timeout, 3% context overflow, 2% server error, 2% JSON parse. Total ~20% failure rate across 50 API calls. Without Helix: 14 tasks failed due to infrastructure. 36 completed, 34 passed. 68% pass@1. With Helix: same seed, same failures. PCEC repaired 12 of 14. 48 completed, 45 passed. 90% pass@1. The 2 it couldn't fix: one genuine context overflow, one persistent server outage. We report both honestly."
LIKELYBut you injected the failures yourself — isn't that cheating?›
"We inject at realistic rates based on real-world API reliability data. Over 50 sequential API calls, the probability of hitting at least one failure is ~22% — that's math, not fiction. We use a fixed seed so anyone can reproduce our exact results — same seed 42, same failures, every time. Both runs see identical failures. The ONLY variable is whether Helix catches them. That's a controlled experiment, not cheating."
If they push harder: "Would you prefer we waited for OpenAI to actually go down during our demo? We simulate what inevitably happens in production."
MEDIUMWhy HumanEval? Isn't that outdated?›
"HumanEval is a proof of concept for the repair engine — it's the most universally recognized coding benchmark, so everyone can calibrate the numbers. We're not claiming to improve the model's intelligence. We're showing that infrastructure reliability matters. For production, SWE-bench Verified is the next benchmark — 500 real GitHub issues, multi-file patches, much higher failure rate. Helix's value would be even more dramatic there. We chose HumanEval for the hackathon because it's fast, cheap, and everyone knows it."
MEDIUMWhat does HumanEval have to do with payments?›

"Nothing directly. HumanEval doesn't test payments — it tests our PCEC repair engine. The payment scenarios on our dashboard show WHAT Helix fixes. The benchmark shows that the repair mechanism WORKS in a real API environment. Same engine, two domains: - Payment failures: insufficient balance → swap_currency - Infrastructure failures: 429 rate limit → backoff_retry Both go through the same Perceive→Construct→Evaluate→Commit loop. HumanEval proves the engine is reliable. Payment Lab proves the engine knows payments."

"GPT-4o-mini's HumanEval score is publicly known — around 86-87% under clean conditions. That gives evaluators a baseline to calibrate against. Claude's HumanEval numbers aren't as widely published. And we're not trying to showcase a model — we're showcasing infrastructure. Using a well-known mid-tier model makes the improvement more credible."

🧬New Scenarios & Advanced Architecture5 Q&As
LIKELYWhat about Token Pause? How do you handle that?›
"TIP-20 spec allows issuers to pause any token — like Circle's USDC freeze. If USDC gets paused mid-transfer, all transfers revert. Helix detects token-paused error → queries TIP-20 Factory for alternative USD stablecoins → swaps to unpaused token (e.g. USDT) → retries. Gene stores which tokens are unstable. This is scenario 11."
MEDIUMWhat happens when a fee sponsor runs out of gas?›
"Tempo supports fee sponsorship — a third party pays gas on behalf of the agent. If the sponsor balance hits zero, the agent's tx silently fails. Helix catches the sponsor-insufficient error → switches to self-pay mode (gas from agent's own wallet, cost ~$0.0001) → retries. Gene stores fallback preference. This is scenario 12."
LIKELYWhat if you encounter a failure type you've never seen?›
"Three layers of defense. First, the error space is bounded — MPP defines 7 error codes, we added 5 Tempo-specific ones. That covers 95%+ of cases. For the remaining 5%: LLM Fallback. When Gene Map returns a miss, PCEC sends the full failure context — error code, message, wallet state, protocol context — to an LLM. The LLM generates repair candidates. We validate them in a sandbox before execution. If successful, the repair becomes a new Gene Capsule tagged with source: 'llm' and low initial confidence. Over time, LLM-generated Genes that keep working get promoted to high confidence. It's a trust ladder — PCEC Genes are trusted immediately, LLM Genes earn trust through production validation."
DEEPHow does the LLM Fallback affect your costs?›
"LLM calls only happen on Gene Map misses — which after the first few weeks of operation will be extremely rare, maybe 1-2% of failures. The other 98% are handled by cached Genes at zero LLM cost. And each LLM call creates a new Gene, so the same unknown failure never triggers an LLM call twice. The cost converges to near-zero over time. That's the Gene Map network effect applied to LLM costs."
MEDIUMDo Gene strategies go stale? What about protocol upgrades?›
"Two mechanisms. Short-term: LLM Fallback handles new error codes from protocol upgrades by generating new Genes on the fly. Long-term: we're building evolutionary parameter optimization. Each Gene has parameters — slippage tolerance, split count, wait time. These need different values under different market conditions. Evolution algorithms search the parameter space continuously, so Genes adapt to changing conditions automatically. The Gene Map isn't static — it's a living knowledge base that both grows (via LLM) and sharpens (via evolution)."
