Judge Cheatsheet — Additional Q&As

Prepared answers for likely judge questions about LLM Fallback, Evolution, and Failure Philosophies.

Technical Deep Dives

LIKELYWhat if you encounter a failure type you've never seen before?
Three layers of defense. First, the error space is bounded — MPP defines 7 error codes, we added 3 Tempo-specific ones. That covers 95%+ of cases. For the remaining 5%: we have an LLM Fallback path. When Gene Map returns a miss, PCEC sends the full failure context — error code, message, wallet state, protocol context — to an LLM. The LLM generates repair candidates. We validate them in a sandbox before execution. If successful, the repair becomes a new Gene Capsule tagged with source: 'llm' and low initial confidence. Over time, LLM-generated Genes that keep working get promoted to high confidence. It's a trust ladder — PCEC Genes are trusted immediately, LLM Genes earn trust through production validation.
MEDIUMDo Gene strategies go stale? What about protocol upgrades?
Two mechanisms. Short-term: LLM Fallback handles new error codes from protocol upgrades by generating new Genes on the fly. Long-term: we're building evolutionary parameter optimization. Each Gene has parameters — slippage tolerance, split count, wait time. These need different values under different market conditions. Evolution algorithms search the parameter space continuously, so Genes adapt to changing conditions automatically. The Gene Map isn't static — it's a living knowledge base that both grows (via LLM) and sharpens (via evolution).
DEEPWhat's the difference between Helix "waiting" and just "giving up"?
Giving up means propagating the error and forgetting about it. Intelligent wait means: I recognize this failure can't be fixed on-chain right now — maybe an address is blacklisted — so I don't waste gas retrying. Instead, I set a watcher on the policy state. The moment the condition changes, I auto-resume. The Gene Capsule for a "wait" scenario stores: what we're waiting for, how to detect the change, and what to do when it happens. It's not giving up — it's scheduled recovery with a trigger condition.
MEDIUMYou mentioned four failure philosophies. Can you explain?
Sure. Most people think self-healing means retry. We have four modes: 1. Active Repair — detect, generate candidates, score, execute. This is 70% of cases. Swap currency, renew session, refresh nonce. 2. Compliance Reroute — regulatory block detected, find an alternative compliant path. Not a retry — a reroute. 3. Temporal Awareness — the failure happened because conditions changed DURING execution. We detect the timing, not just the error. Phase 2. 4. Intelligent Wait — some failures can't be fixed on-chain. We recognize this, stop trying, and set a watcher to auto-resume when conditions change. This is the one that separates us from every retry library.

Business & Market

DEEPHow does the LLM Fallback affect your costs?
LLM calls only happen on Gene Map misses — which after the first few weeks of operation will be extremely rare, maybe 1-2% of failures. The other 98% are handled by cached Genes at zero LLM cost. And each LLM call creates a new Gene, so the same unknown failure never triggers an LLM call twice. The cost converges to near-zero over time. That's the Gene Map network effect applied to LLM costs.
